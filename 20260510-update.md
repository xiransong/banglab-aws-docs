# Rules Update

## Shared AWS Account Resource Ownership Rules

Because our AWS Organization has a limited number of accounts, multiple researchers may share the same AWS member account. To avoid interfering with each other’s resources, we use an **owner-based resource isolation rule**.

Each user has a canonical owner name, for example:

```text
xiransong
alicezhang
bobchen
```

This owner name is used consistently in IAM Identity Center, EC2 tags, EBS volume tags, and S3 bucket names.

---

## 1. Required `Owner` tag for EC2 and EBS

All user-created EC2 instances and EBS volumes must be tagged with:

```text
Owner=<username>
```

For example:

```text
Owner=xiransong
```

Users can only start, stop, reboot, terminate, attach, detach, modify, or delete resources whose `Owner` tag matches their own IAM Identity Center owner value.

Example EC2/EBS tags:

```text
Owner=xiransong
Name=xiransong-dev
```

```text
Owner=xiransong
Name=xiransong-persistent-ebs
```

This prevents one user from accidentally terminating another user’s instance or deleting another user’s persistent EBS volume.

---

## 2. S3 bucket naming rule

Each user should create S3 buckets using the following naming convention:

```text
banglab-<username>-*
```

For example:

```text
banglab-xiransong-data
banglab-xiransong-scratch
banglab-alicezhang-data
```

Users can only access S3 buckets that match their own prefix:

```text
banglab-${Owner}-*
```

So `xiransong` can access:

```text
s3://banglab-xiransong-data
```

but not:

```text
s3://banglab-alicezhang-data
```

Legacy personal buckets should be migrated to this naming scheme. For example:

```bash
aws s3 sync s3://xiransong s3://banglab-xiransong-data --copy-props none
```

---

## 3. Launching EC2 instances

When launching an EC2 instance, users must include the required owner tag:

```text
Owner=<username>
```

Example:

```text
Owner=xiransong
Name=xiransong-gpu-dev
```

Instances without the correct `Owner` tag may not be manageable by the user after launch.

---

## 4. Persistent EBS volumes

Persistent EBS volumes should also use the same owner tag:

```text
Owner=<username>
Name=<username>-persistent-ebs
```

Example:

```bash
aws ec2 create-tags \
  --resources vol-xxxxxxxxxxxxxxxxx \
  --tags Key=Owner,Value=xiransong Key=Name,Value=xiransong-persistent-ebs \
  --region us-east-1
```

Only the owner can attach, detach, modify, snapshot, or delete the volume.

---

## 5. Permission model

The `EC2-GPU-Operator` permission set now uses **tag-based access control**.

In practice:

```text
A user can manage their own EC2 instances.
A user can manage their own EBS volumes.
A user can access their own S3 buckets.
A user cannot terminate another user’s EC2 instance.
A user cannot access another user’s S3 bucket.
A user cannot delete another user’s EBS volume.
```

The matching rule is:

```text
Resource Owner tag == IAM Identity Center Owner attribute
```

For S3, the matching rule is based on bucket name:

```text
banglab-<Owner>-*
```

---

## 6. Admin recovery rule

Admins should keep a separate administrator permission set. If a resource is accidentally created without an `Owner` tag, normal users may not be able to manage it. In that case, an admin can add or fix the tag.

Example:

```bash
aws ec2 create-tags \
  --resources i-xxxxxxxxxxxxxxxxx \
  --tags Key=Owner,Value=xiransong Key=Name,Value=xiransong-dev \
  --region us-east-1
```

---

A compact version:

> In shared AWS accounts, every researcher is assigned an `Owner` value through IAM Identity Center. EC2 instances and EBS volumes must be tagged with `Owner=<username>`, and S3 buckets must be named `banglab-<username>-*`. The `EC2-GPU-Operator` permission set uses this owner value to restrict users to their own resources. This prevents users from accidentally terminating each other’s EC2 instances, deleting each other’s EBS volumes, or accessing each other’s S3 buckets.
